# RICOVER Usage Guide

## Prerequisites

- Rust toolchain (edition 2024)
- Local clone of [isla](https://github.com/rems-project/isla) (isla-lib is a path dependency)
- A compiled `.ir` file from the RISC-V Sail model (generated by `isla-sail`)
- Z3 or Eldarica for solving the output CHC

## Building

```bash
cd Implementation/RICOVER
cargo build
```

## Commands

### `translate-ir` — Extract instruction semantics as CHC

Reads an Isla IR file (`.ir`) and translates specified function bodies into CHC definitions in SMT-LIB2 HORN format.

```bash
# Reads an Isla IR file (`.ir`) and 
# translates specified function bodies into 
# CHC definitions in SMT-LIB2 HORN format.
cargo run -- translate-ir \
  -i path/to/riscv64.ir \
  -o instructions.smt2 \
  -f execute
```

- `-i` — path to the `.ir` file (e.g., `riscv64.ir` from isla-sail)
- `-o` — output `.smt2` file
- `-f` — function names to translate (can be repeated: `-f execute -f rX -f wX`)

The output includes the CHC stdlib (state/register/memory definitions) followed by the translated instruction semantics.

### `check-equiv` — Compare two assembly functions

Parses two RISC-V assembly files and emits a CHC equivalence query.

```bash
# Parses two RISC-V assembly files and emits 
# a CHC equivalence query.
cargo run -- check-equiv \
  --before foo_O0.s \
  --after foo_O2.s \
  -f foo \
  -o query.smt2
```

- `--before` — assembly file with unoptimized code
- `--after` — assembly file with optimized code
- `-f` — function name to compare
- `-o` — output `.smt2` file

### Solving

Feed the output to a CHC solver:

```bash
z3 query.smt2
```

- **unsat** — the two programs are semantically equivalent (no initial state produces different results)
- **sat** — counterexample exists (the solver can provide a model showing divergent initial state)

## How it works

1. The RISC-V Sail specification is compiled to Isla IR (`.ir` file) by `isla-sail`
2. `translate-ir` parses the IR and walks instruction execute clauses (e.g., ADDI, LOAD), emitting CHC relations that encode state transitions over `(PC, Regs, Mem)`
3. `check-equiv` parses two `.s` files, chains instruction-level CHC relations with state threading (`S0 → S1 → ... → Sn`), and emits a query asserting the final states differ
4. A CHC solver (Z3/Spacer, Eldarica) checks satisfiability — unsat means the optimization preserved semantics

## CHC stdlib

The standard library (`chc_stdlib/stdlib.smt2`) defines:

- **State** — `(PC: BitVec 64, Regs: Array(BitVec 5 → BitVec 64), Mem: Array(BitVec 64 → BitVec 8))`
- `get_reg(regs, idx)` — read register, returns 0 for x0
- `set_reg(regs, idx, val)` — write register, discards writes to x0
- `read_mem_word` / `read_mem_dword` — little-endian loads (4/8 bytes)
- `write_mem_word` / `write_mem_dword` — little-endian stores (4/8 bytes)
- Register ABI constants — `reg_zero`, `reg_ra`, `reg_sp`, `reg_s0`, `reg_a0`

## Generating the .ir file

If you need to regenerate the IR from the Sail model:

```bash
# In the isla repo
cd isla
cargo build --release
# Compile the RISC-V Sail model to IR
target/release/isla-sail 
-i sail-riscv/model 
-o riscv64.ir
```

The minimal `riscv.ir` in `Tools/Sail/Data/` covers ITYPE (addi) and LOAD instructions. The full `riscv64.ir` covers all RV64 instructions.
